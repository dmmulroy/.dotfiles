# Goal
Extend Pi's `webfetch` capability so it can use a selected Helium browser profile's authenticated session state, while keeping the existing `webfetch` tool model intact.

## Key product decision
Do **not** add a separate browser automation flow and do **not** launch a browser process (headless or otherwise).

Instead:
- keep using the same `webfetch` tool
- when a Helium profile is selected, `webfetch` should perform normal HTTP fetches but with auth/session data sourced from that Helium profile
- prefer **CDP** when available
- fall back to **on-disk persisted cookies** when CDP is unavailable

## UX decisions

### Footer indicator
Use footer status, not per-tool result annotation.

Desired footer states:
- `web: public`
- `web: Helium/<display-name>`

Notes:
- If a Helium profile is selected, the assumption is that authenticated fetching is always desired.
- No extra `[auth]` suffix is needed.
- No need to show profile usage in each `webfetch` tool result; footer status is enough.

### Single command
Use only one command for now:
- `/web-profile`

This command should:
- show current selection
- allow switching back to `Public web`
- allow picking a discovered Helium profile
- persist the selection
- update footer status immediately

Do **not** add extra commands like `/web-profile clear` or `/web-profile status` for now.

## Technical decisions

### Keep `webfetch` as the main tool
No separate authenticated-web or browser-fetch tool.

Behavior:
- if active selection is `public`, `webfetch` behaves exactly as it does today
- if active selection is a Helium profile, `webfetch` injects auth/session state before making the request

### Auth source priority
1. **CDP first**
   - If Helium is running and its DevTools endpoint is reachable, use CDP to retrieve the freshest relevant cookies for the request URL.
   - This is especially important for session cookies or cookies newer than disk state.
2. **Disk cookie fallback**
   - Read persisted cookies from the selected Helium profile on disk when CDP is not available.

### Important non-goals
- Do not launch a browser process.
- Do not do headless browsing.
- Do not build a rendered-page fetch path as the primary implementation.
- Do not add UI for user-facing fetch modes like `auto | browser | http`.
  - Any strategy choice (`cdp` vs `disk-cookies`) should remain an internal implementation detail.

```ts
type ActiveWebIdentity =
  | { kind: "public" }
  | { kind: "helium"; profileId: string; displayName: string };
```

### Internal resolved auth context
This is useful for debugging/logging but should remain internal rather than becoming a user-facing mode toggle.

```ts
type ResolvedAuthContext = {
  identity: "public" | "helium";
  strategy: "none" | "cdp" | "disk-cookies";
  cookieCount?: number;
};
```

## State persistence decision
Persist the selected web identity as tool/extension state, not as a started implementation task.

Preferred behavior:
- store selected active profile in extension-managed state so it survives reloads/restarts
- restore it on `session_start`
- refresh footer status immediately

This feels more like environment/tool configuration than conversation content.

## Scope boundaries / expectations
This design is primarily for authenticated HTTP requests using existing browser session state.

Best fit:
- sites authenticated with cookies
- persisted Chromium/Helium session state
- session cookies retrievable through CDP or disk

Potential limitations for later consideration:
- sites relying primarily on localStorage/sessionStorage tokens
- service-worker-mediated auth flows
- custom JS request signing

These are not blockers for v1; v1 should focus on cookies/session reuse.

## Summary of final agreed direction
- Keep the existing `webfetch` tool.
- Add a single `/web-profile` command.
- Show selected profile in footer as `web: public` or `web: Helium/<name>`.
- Selecting a Helium profile means authenticated fetches are always desired.
- Use **CDP first** for live cookie/session retrieval.
- Fall back to **disk cookies** from the selected Helium profile.
- Do **not** launch browsers.
- Do **not** add a separate browser-backed fetch mode/tool.
- Do **not** add extra profile management commands yet.
