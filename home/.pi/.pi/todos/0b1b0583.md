# Work items

## 1) Fix CDP close/hang behavior so auth resolution cannot wedge webfetch

### Problem
In `agent/extensions/web-tools/auth-cdp.ts`, `callBrowserCdp()` currently sets `ws.onclose = () => { cleanup(); }` and does not reject the promise on an unexpected close.

That can leave the CDP request promise pending forever if the socket opens and then closes without delivering an `error` or a matching reply message. Because `webfetch` waits for header construction before issuing the HTTP request, this can hang the whole tool.

### Required implementation
1. Update `callBrowserCdp()` in `agent/extensions/web-tools/auth-cdp.ts` so the promise always settles exactly once.
2. Track completion explicitly, e.g. with a `settled` boolean and helper `finishResolve` / `finishReject` wrappers.
3. Treat an unexpected `close` before response completion as an error and reject with a useful message, e.g. `Helium CDP connection closed before reply for Storage.getCookies`.
4. Ensure timeout, `error`, malformed response, and `close` all reject only once.
5. Keep the existing “do not launch a browser” rule intact.

### Strong recommendation
Thread cancellation/timeout into auth resolution so a stuck auth source cannot outlive the tool timeout.

Suggested approach:
- Introduce an optional `signal?: AbortSignal` in auth resolution plumbing:
  - `resolveRequestAuth(...)`
  - `getCookiesFromCdp(...)`
  - possibly `getCookiesFromProfileDb(...)` if practical
- In `webfetch.ts`, pass the composed operation signal from `createOperationSignal()` into auth resolution.
- In `auth-cdp.ts`, if `signal` aborts:
  - close the WebSocket
  - reject promptly

If threading a signal all the way through is too invasive, at minimum ensure CDP has its own bounded timeout and cannot hang indefinitely on close.

### Tests to add
In `agent/extensions/web-tools/test/auth-cdp.test.ts` and/or `test/auth.test.ts`:
- Add a test where the CDP socket closes before any response arrives; expect auth resolution to fail fast.
- Add a test that the failure path falls back cleanly to disk cookies.
- If signal support is added, add a test showing an aborted signal cancels the CDP auth attempt.

## 2) Fix CDP auth so cookies are scoped to the selected Helium profile

### Problem
In `agent/extensions/web-tools/auth-cdp.ts`, `fetchBrowserCookiesFromCdp()` calls:
- `Storage.getCookies` with `{}`

The selected `/web-profile` is a specific Helium profile, but no `browserContextId` or equivalent profile scoping is currently passed to CDP. That means the CDP path may return the wrong profile’s cookies or only the default context’s cookies in multi-profile setups.

### Required investigation
Review Helium/Chromium behavior for profile-to-browser-context mapping. Use the CDP docs above as a starting point.

### Implementation goal
Make the CDP auth path profile-aware.

### Acceptable implementation paths
Choose one of these, in order of preference:

1. **Preferred:** Determine the selected profile’s browser context and pass `browserContextId` to `Storage.getCookies`.
   - You may need an additional CDP query if Chromium exposes profile/context metadata separately.
   - Keep the request read-only.

2. **Fallback-safe:** If you cannot reliably map the selected Helium profile to a CDP browser context, do **not** use unscoped browser-wide cookies for authenticated requests.
   - In that case, detect that CDP scoping is unavailable and skip/fail the CDP path so disk cookies become the first usable auth source.
   - This is safer than injecting possibly wrong-profile cookies.

### Important constraints
- Do not launch Helium.
- Do not add another tool.
- Keep `/web-profile` as the only selector.
- Preserve footer-only status behavior.

### Tests to add
In `agent/extensions/web-tools/test/auth-cdp.test.ts` and/or `test/auth.test.ts`:
- Add tests that assert the selected profile is actually respected by the CDP path.
- If you implement a safe skip when scope cannot be determined, test that the code falls back to disk cookies instead of using unscoped browser-wide cookies.

## 3) Stop silently degrading to unauthenticated/public behavior when auth resolution operationally fails

### Problem
Current behavior in `agent/extensions/web-tools/auth.ts` and `agent/extensions/web-tools/auth-cookies.ts` can make a Helium-selected session appear to succeed while actually sending no authenticated cookies:
- CDP throws → falls back to disk cookies
- disk snapshot/query/decrypt problems may get swallowed or reduced to empty-cookie output
- result becomes `strategy: "disk-cookies"` with `cookieCount: 0`

That hides the difference between:
- no cookies match the URL
- auth source unavailable / broken
- cookie DB unreadable
- decryption failed

### Required implementation
Refactor auth resolution to distinguish **operational failure** from **valid empty result**.

Suggested approach:
1. Introduce a richer result shape in `agent/extensions/web-tools/types.ts` and `auth.ts`.
   - For example, expand auth context or create internal result types that can represent:
     - `source attempted`
     - `source failed`
     - `source unavailable`
     - `matched cookie count`
2. In `auth-cookies.ts`, avoid broad silent fallbacks that erase failure causes.
   - `queryCookieSnapshot()` currently returns `{ rows: [], metaVersion: 0 }` on any error.
   - Change it so callers can distinguish “DB missing” from “snapshot/query failed”.
3. In `resolveRequestAuth()`:
   - if CDP fails and disk cookies also fail operationally, do **not** silently pretend auth succeeded with zero cookies.
   - Either:
     - throw a clear error, or
     - return an explicit auth-unavailable state that `webfetch` surfaces clearly.

### Preferred behavior
Because the product rule says that selecting a Helium profile means authenticated fetching is desired, prefer **failing clearly** when both auth sources are unavailable/broken.

However, do not treat “source succeeded but there were simply no matching cookies for this URL” as an operational failure.

### UX guidance
If you choose explicit surfaced failure rather than throwing immediately, make sure any surfaced message is careful not to leak:
- cookie values
- keychain secrets
- raw DB paths beyond what is already normal for local diagnostics

### Tests to add
In `agent/extensions/web-tools/test/auth.test.ts` and `test/webfetch.test.ts`:
- Add a test where CDP fails and disk cookie access also fails; expect a clear auth failure rather than silent zero-cookie downgrade.
- Add a test where auth sources succeed but there are genuinely no matching cookies for the target URL; that should remain a non-error.
- Add a test where disk cookie DB exists but decryption fails; ensure the failure semantics are the chosen intended behavior.

## 6) Add explicit handling or exclusion for partitioned cookies (CHIPS)

### Problem
The current auth cookie model in `agent/extensions/web-tools/types.ts` only tracks:
- name
- value
- domain
- path
- secure
- httpOnly
- expiresAt
- hostOnly

It does not model partition metadata from modern Chromium cookies. That means partitioned cookies could be mishandled if returned by CDP or read from the DB.

### Required implementation
Choose one of these:

1. **Conservative approach (recommended if quick hardening):**
   - Detect partitioned cookies from CDP and/or DB
   - Exclude them from injection for now
   - Add comments/docs/tests stating that partitioned cookies are intentionally not yet supported

2. **Full support approach:**
   - Extend `AuthCookie` in `types.ts` with partition metadata
   - Parse relevant fields from CDP and the cookie DB schema if available
   - Update matching logic in `auth.ts` so partitioned cookies are sent only when the partition context is appropriate

### Guidance
If you are not deeply certain about correct CHIPS semantics in this extension, prefer explicit exclusion over incorrect broad injection.

### Tests to add
- Add at least one regression test showing partitioned cookies are not accidentally treated as normal cookies.
- If full support is implemented, add URL/context matching tests for that behavior.
