# Goal

Redesign the `agent/extensions/web-tools` authenticated `webfetch` stack from scratch around a simpler, more deterministic, more robust model that maximizes **seamless reuse of existing Chromium/Helium login state** while minimizing CDP prompts and avoiding unnecessary complexity.

This redesign should incorporate the main lessons from the recent investigation, especially:

- current `webfetch` auth should stay fundamentally **HTTP-based**, not browser-automation-based
- **disk-backed browser profile state is the persistent source of truth**
- Pi Web Access succeeds largely because it is **deterministic, narrow, and robust about disk cookie extraction**, not because it has a complex auth orchestration system
- repeated user friction comes primarily from **disk-read failures, profile ambiguity, and unnecessary CDP refreshes**, not from lack of more auth layers
- challenge-protected sites may still fail even with correct cookies; the redesign should improve correctness and DX without pretending raw HTTP can fully emulate a browser

# Redesign principles

## 1. Disk is the persistent source of truth

Do **not** add another persisted auth snapshot/cache layer.

Rationale:
- Chromium/Helium profile state on disk is already the canonical persisted auth store
- duplicating decrypted cookies into another app-managed persistence layer increases security risk and invalidation complexity
- a second persisted layer does not solve the fundamental raw-HTTP-vs-browser gap

Resulting policy:
- **persistent state** = browser profile on disk
- **ephemeral acceleration** = in-memory session cache only
- **interactive refresh** = CDP only when necessary

## 2. Deterministic, narrow source resolution

Borrow the strongest Pi Web Access lesson:
- prefer known browser/profile locations
- use explicit profile selection
- use simple, deterministic fallback order
- avoid dynamic “smart” behavior that is hard to reason about

Resulting source order:
- default authenticated source order: **disk → CDP**
- if a fresh successful CDP jar already exists in memory, prefer **CDP → disk** for a short TTL
- no negative caching for failed CDP attempts

## 3. Split profile management from cookie extraction from request auth

Current/auth-adjacent logic should be decomposed into sharply separated responsibilities.

### A. Profile registry layer
Responsible for:
- discovering supported Chromium-family profile roots
- enumerating browser + profile identities
- marking which profiles have readable cookie DBs
- marking whether CDP appears reachable
- resolving the selected profile deterministically

### B. Disk cookie extraction layer
Responsible for:
- copying `Cookies` DB to temp
- copying WAL/SHM sidecars
- reading meta version
- decrypting cookie values
- returning normalized cookie jar records

### C. CDP cookie refresh layer
Responsible for:
- best-effort live-cookie refresh only
- strict profile safety checks before use
- converting CDP cookies into same normalized jar model

### D. Request auth layer
Responsible for:
- deciding which source to ask for cookies
- selecting cookies for the outgoing URL/request context
- building the `Cookie` header
- exposing auth metadata for debug/details

### E. In-memory session cache layer
Responsible for:
- short TTL cache of normalized cookie jars keyed by selected profile + source
- cache invalidation on profile/session change
- no disk persistence

This separation should make each failure mode obvious.

## 4. Cookie correctness matters more than source cleverness

The rebuild should invest more in **cookie selection correctness** than in adding more auth strategy branches.

### Expand the cookie model
Replace the current simplified cookie shape with a fuller normalized Chromium cookie record.

Suggested normalized type:

```ts
interface NormalizedCookie {
  name: string;
  value: string;
  domain: string;
  path: string;
  secure: boolean;
  httpOnly: boolean;
  hostOnly: boolean;
  expiresAt?: number;
  sameSite?: "Strict" | "Lax" | "None";
  sourcePort?: number;
  sourceScheme?: "NonSecure" | "Secure" | "Unknown";
  partitionKey?: string;
}
```

### Selection policy
Implement explicit request-side cookie matching for:
- expiry
- secure-only behavior
- host-only vs domain cookies
- RFC-ish path matching
- SameSite treatment with a conservative navigation model

For `webfetch`, treat the fetch as closest to a **top-level document navigation GET** unless a better request-context abstraction is introduced.

### Partitioned cookies
Do **not** silently “support” partitioned cookies with incomplete logic.

Initial policy:
- parse/store partition metadata if available
- exclude partitioned cookies from injection until matching semantics are implemented correctly
- surface debug metadata showing how many cookies were excluded for partitioning reasons

This is safer than pretending they work.

## 5. CDP should be a refresh path, not a peer persistence layer

CDP use should be reframed as:
- a **live refresh mechanism** when disk could not provide usable auth
- not a general equal-weight auth backend

### CDP trigger policy
Only try CDP when one of these is true:
1. disk cookie DB missing/unreadable
2. cookie decryption failed
3. request using disk cookies returned `401` or `403`

### CDP success handling
On successful CDP cookie retrieval:
- normalize cookies into the same jar format as disk cookies
- place in session cache with a somewhat longer TTL than disk-read results
- use it immediately for retry
- prefer it for a short period after success

### CDP failure handling
- do not negatively cache CDP failure
- do not repeatedly spam CDP inside one request
- at most one CDP refresh attempt per request

## 6. Browser-profile UX should be simple and stable

Follow the spirit of Pi Web Access here.

### Selected profile model
Persist a single stable selected identity, e.g.:

```ts
interface SelectedBrowserProfile {
  browser: "helium";
  userDataDir: string;
  profileId: string;
  displayName: string;
}
```

### UX rules
- if selected profile still exists, keep using it
- if it disappears, fall back to public mode and surface clear status
- do not silently hop between profiles
- avoid prompting if the last selected profile is still resolvable

### Discovery order
Use a deterministic browser/profile discovery order, similar in spirit to Pi Web Access:
- known user-data roots first
- known profile naming conventions
- stable sorting rules

## 7. Request headers should look more like a browser navigation

Raw HTTP will never equal a browser, but request shape still matters.

Replace the minimal header builder with a more intentional navigation-oriented profile.

Suggested default header set for HTML-like fetches:

```ts
User-Agent: <browser-like UA>
Accept: text/html,application/xhtml+xml,application/xml;q=0.9,image/avif,image/webp,*/*;q=0.8
Accept-Language: en-US,en;q=0.9
Cache-Control: max-age=0
Upgrade-Insecure-Requests: 1
Sec-Fetch-Dest: document
Sec-Fetch-Mode: navigate
Sec-Fetch-Site: none
Sec-Fetch-User: ?1
```

Optional heuristics:
- for redirects within same origin, adjust `Sec-Fetch-Site`
- keep these headers static/simple initially rather than building an overcomplicated browser simulator

Do **not** overfit to Cloudflare. Keep headers generic.

## 8. Debuggability should expose source decisions, not implementation noise

Every failed authenticated fetch should make it obvious:
- selected profile
- attempted source order
- which source actually produced cookies
- cookie counts injected
- cookies excluded because of expiry/SameSite/partitioning
- whether request retried with CDP
- whether fallback UA/header profile was used

This should be part of structured details/debug output, not only ad hoc error strings.

# Proposed rebuild: file/module architecture

## Keep or adapt

- `profiles.ts` — keep concept, simplify and harden discovery rules
- `profile-ui.ts` — keep, but ensure it surfaces disk-readability and CDP reachability clearly
- `profile-state.ts` — keep stable selected-profile persistence
- `webfetch.ts` — keep as tool entry point, but simplify its auth orchestration surface

## Replace with a cleaner layout

### `browser-profiles.ts`
Responsibility:
- known browser roots
- profile enumeration
- stable discovery ordering
- lightweight readiness metadata

### `cookie-db.ts`
Responsibility:
- temp DB snapshot creation
- WAL/SHM copying
- SQLite query helpers
- DB meta/version reads

### `cookie-decrypt.ts`
Responsibility:
- safe-storage/keychain password lookup
- Chromium cookie decryption
- meta-version-specific host digest handling

### `cookie-jar.ts`
Responsibility:
- normalized cookie model
- URL/request-context matching
- cookie sorting and header building
- exclusion accounting/debug stats

### `auth-disk.ts`
Responsibility:
- load normalized jar from selected profile’s disk state

### `auth-cdp-refresh.ts`
Responsibility:
- safe CDP access
- selected-profile scope checks
- normalized jar conversion

### `auth-cache.ts`
Responsibility:
- in-memory jar cache keyed by `userDataDir + profileId + source`
- TTL policy
- cache invalidation helpers

### `request-auth.ts`
Responsibility:
- main source ordering policy
- request-time auth resolution
- one-shot refresh rules
- auth debug context

### `auth-types.ts`
Responsibility:
- keep auth types isolated from generic tool types

This structure reduces the current cross-file coupling.

# Request flow redesign

## Public request path

If no selected authenticated profile:
- build public browser-like headers
- fetch normally
- no auth-specific work

## Authenticated request path

### Phase 1: resolve auth source
1. resolve selected profile
2. choose preferred source order
   - default: `disk`, then `cdp-refresh`
   - if fresh successful CDP cache exists: `cdp-refresh`, then `disk`
3. load normalized jar from first available source
4. select cookies for target URL using request-context matching
5. build `Cookie` header
6. emit auth debug metadata

### Phase 2: perform request
1. send request with browser-like navigation headers + cookies
2. if response is OK, return
3. if response is `401` or `403` and auth came from disk:
   - perform one CDP refresh attempt
   - retry once with refreshed jar
4. if still failing, return failure with structured auth debug details

### Explicit non-goals
- no persistent duplicate auth snapshot store
- no browser launch
- no generalized storageState/localStorage replay
- no Cloudflare-specific special cases

# Cache design

## Only in-memory cache

Cache object key:

```ts
${userDataDir}::${profileId}::${source}
```

Cache value:
- normalized cookie jar
- source
- loadedAt
- expiresAt
- diagnostics (counts, excluded partitioned count, etc.)

## TTLs

Suggested initial TTLs:
- disk jar cache: 15–30 seconds
- CDP-refreshed jar cache: 60–120 seconds

Rationale:
- disk reads are cheap but not free
- CDP success should reduce repeated prompts for a short period
- TTLs remain short enough to avoid long-lived stale auth behavior

## Invalidation

Clear cache on:
- Pi session start
- selected profile change
- selected identity change
- explicit auth failure that proves jar is stale/unusable

Do not clear all cache on every ordinary error.

- selected profile resolution stable across refreshes
- missing selected profile falls back to public mode

### Disk DB reading

- domain cookies support subdomains

- expired cookies excluded
- SameSite handling behaves as expected
- partitioned cookies excluded and counted

### Request auth policy
- public mode skips auth entirely
- disk-first succeeds without CDP
- disk unreadable triggers CDP refresh attempt
- disk-auth 401/403 triggers one CDP retry
- successful CDP retry is cached briefly
- failed CDP retry is not negatively cached

- `webfetch` with selected authenticated profile injects expected cookies
- redirect chain preserves request-time cookie selection per target URL
- auth debug metadata reflects actual source path used
- profile/session change clears in-memory cache

# Recommended final shape

If rebuilding from scratch, the target system should feel like this:

- **profile selection is explicit and stable**
- **disk-backed Chromium state is the canonical auth source**
- **cookie extraction from disk is extremely robust**
- **cookie matching is more correct than it is today**
- **CDP is only a low-frequency refresh escape hatch**
- **all caches are in-memory only**
- **request headers are modestly more browser-like**
- **failures explain exactly what source/path was attempted**

That design best aligns with the goals:
- minimal prompts
- minimal intervention
- maximum reuse of existing logged-in browser state
- clearer architecture than the current auth stack
- better lessons applied from Pi Web Access without overfitting to its narrower Gemini-specific implementation

## Additional requirement: preserve current profile-selection UX/DX

The redesign should **keep the existing UI/UX/DX around profile selection** unless there is a very strong reason to change it.

Specifically:
- preserve the current `/web-profile` command flow
- preserve the current general picker/status experience
- preserve the notion of an explicitly selected active profile/identity
- do not regress discoverability or add extra user steps
- internal architecture may change substantially, but the user-facing profile-selection interaction should remain familiar

Implementation guidance:
- prefer swapping in the new profile registry / auth backends **behind** the existing picker and persisted identity model
- only change profile UI if required to expose materially useful readiness information (for example, disk DB available vs unavailable, CDP reachable vs unreachable), and even then keep the same overall interaction model
- avoid replacing the current explicit profile-selection UX with config-only or auto-pick-only behavior
